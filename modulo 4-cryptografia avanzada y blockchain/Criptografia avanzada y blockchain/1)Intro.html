Iniciamos nuestro viaje al corazón mismo de la seguridad digital. Antes de entender tecnologías complejas como Blockchain, debemos dominar los ladrillos fundamentales que las construyen: la criptografía y sus bases matemáticas. Esta primera semana está dedicada a desmitificar conceptos que a menudo se usan incorrectamente en la industria, como "cifrado", "hash", "firma digital" y "claves criptográficas".

La criptografía moderna no se trata únicamente de proteger secretos militares; se trata de garantizar que un mensaje no ha sido alterado durante la transmisión (Integridad), que el remitente es realmente quien dice ser (Autenticación), y que este no pueda negar posteriormente haber enviado el mensaje (No Repudio). Estos principios son universales y aplican desde un simple chat seguro hasta una transacción de Bitcoin de millones de dólares.

🎯 Objetivo Principal de la Semana: Comprender la diferencia arquitectónica y funcional entre criptografía simétrica y asimétrica, dominar las operaciones matemáticas fundamentales que las hacen posibles, y saber cuándo aplicar cada enfoque en escenarios reales de tecnología de la información.

🔍 Comparación Detallada: Simétrico vs Asimétrico

Aspecto Criptografía Simétrica Criptografía Asimétrica
Claves Una clave para cifrar y descifrar Par de claves: pública y privada
Velocidad Muy rápido (100-1000x más rápido) Lento (costoso computacionalmente)
Distribución de claves Problema difícil (requiere canal seguro) Fácil (clave pública puede compartirse)
Casos de uso principales Cifrado de datos masivos, comunicaciones seguras Intercambio de claves, firmas digitales, autenticación
Algoritmos comunes AES, ChaCha20, DES (obsoleto), 3DES RSA, ECC, Diffie-Hellman, ElGamal
Longitud de clave típica 128-256 bits 2048-4096 bits (RSA) o 256-521 bits (ECC)
Seguridad frente a computación cuántica Resistente (AES-256) Vulnerable (RSA/ECC pueden romperse)
💡 Conclusión práctica: En sistemas modernos, se combinan ambos tipos: RSA/ECC para intercambiar una clave AES, y luego AES para cifrar los datos. Esto es exactamente cómo funciona HTTPS/TLS.

🧮 Fundamentos Matemáticos para Criptografía

La criptografía moderna se basa en problemas matemáticos computacionalmente difíciles. Sin entender estos fundamentos, no se puede comprender realmente cómo funcionan los sistemas seguros.

mod

Aritmética Modular

"Reloj matemático": 15 mod 12 = 3. Fundamental para RSA, Diffie-Hellman y ECC. Las operaciones "vuelven a empezar" después de llegar a un límite (el módulo). Ejemplo: en un reloj de 12 horas, 15 horas = 3 PM.

a^b

Exponenciación Modular

Calcular a^b mod n es fácil incluso con números enormes. Pero encontrar b dado a^b mod n y a es computacionalmente difícil (logaritmo discreto). Esta asimetría es la base de Diffie-Hellman y DSA.

P

Números Primos y Factorización

Multiplicar dos números primos grandes es fácil (p × q = n). Factorizar n para encontrar p y q es extremadamente difícil. RSA se basa en que factorizar números de 2048 bits tomaría miles de años con computadoras actuales.

🎯 Meta de aprendizaje matemático: No necesitas convertirte en matemático, pero debes entender POR QUÉ estos problemas son difíciles y CÓMO esa dificultad crea seguridad.

Cifrado (Encryption)

Transforma información legible (plaintext) en ilegible (ciphertext) usando una clave. Garantiza confidencialidad.

  • Simétrico: AES, ChaCha20, misma clave para cifrar/descifrar
  • Asimétrico: RSA, ECC, par de claves pública/privada
  • Ejemplo: WhatsApp cifra mensajes con Signal Protocol

Hash

"Huella digital" de longitud fija para cualquier dato. Misma entrada → mismo hash. Pequeño cambio → hash completamente diferente. Garantiza integridad.

  • SHA-256: 256 bits, usado en Bitcoin
  • SHA-3: Nueva generación, diferente diseño
  • Ejemplo: Verificar que un archivo ISO no fue corrupto en descarga

HMAC (Hash-based MAC)

Combina hash con clave secreta. Verifica tanto integridad como autenticidad (solo quien conoce la clave pudo generarlo).

  • HMAC-SHA256: Estándar actual para integridad+autenticación
  • Uso: APIs web (tokens JWT), comunicaciones internas seguras
  • Ejemplo: GitHub webhooks firman payloads con HMAC

Firma Digital

Prueba criptográfica de autoría + integridad. Se crea cifrando hash del documento con clave privada. Cualquiera puede verificar con clave pública.

  • Garantiza: No repudio (autor no puede negar autoría)
  • Algoritmos: RSA-PSS, ECDSA, EdDSA
  • Ejemplo: Firmar contratos electrónicos con validez legal

🌍 ¿Dónde se usan estos conceptos en la vida real?

HTTPS/TLS
RSA/ECC para autenticar servidor + intercambiar clave AES, luego AES para cifrar sesión
Mensajería Segura
Signal/WhatsApp: Double Ratchet con X3DH para cifrado E2E + autenticación de claves
Contraseñas
No se cifran, se hashean con salt (Argon2, bcrypt, PBKDF2) para resistir fuerza bruta
Documentos Legales
Firmas digitales PKCS#7 para contratos, actas y certificados con validez legal
Pagos en Línea
EMV tokens, 3D Secure con cifrado simétrico para proteger datos de tarjetas
Bases de Datos
Transparent Data Encryption (TDE) con AES para cifrado en reposo de datos sensibles

Ejercicio Práctico: Análisis de HTTPS/TLS

Objetivo: Entender cómo se combinan criptografía simétrica y asimétrica en el mundo real.

Instrucciones: Abre tu navegador y visita cualquier sitio HTTPS (ej: https://google.com). Haz clic en el candado y examina los certificados.

  1. Identifica el algoritmo de firma del certificado (probablemente RSA o ECDSA)
  2. Busca la longitud de clave pública (2048 bits para RSA, 256 bits para ECC)
  3. Examina la cadena de certificados (jerarquía PKI)
  4. Investiga qué algoritmo de intercambio de claves se usó (RSA, ECDHE)
  5. Identifica el algoritmo simétrico para cifrado de datos (AES_128_GCM, AES_256_GCM, CHACHA20_POLY1305)
💡 Reflexión: ¿Por qué HTTPS usa RSA/ECC solo para autenticar y luego cambia a AES? ¿Qué ventajas tiene esta combinación híbrida?
