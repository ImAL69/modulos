Tema 3.6: Diseño de Arquitectura

Creando la solución conceptual.

1. Arquitectura Híbrida

La mayoría de las soluciones reales son híbridas. Usan una base de datos tradicional para velocidad y datos privados, y una blockchain solo para anclar "pruebas" (hashes) públicas.

2. Gestión de Identidad (IAM)

¿Quién tiene las llaves? Diseña un sistema donde el usuario final no tenga que gestionar claves privadas complejas directamente (Custodial Wallet) o usa soluciones de identidad soberana (Non-Custodial) si el usuario es técnico.

3. Plan de Despliegue

Fase 1: Piloto en Testnet (Costo cero).
Fase 2: Auditoría de Seguridad del Smart Contract.
Fase 3: Despliegue en Mainnet.

🏗️ Arquitectura orientada a riesgo

Diseñar arquitectura es diseñar límites de daño. El objetivo es que un fallo no comprometa todo.

  • Capas independientes.
  • Gestión de claves separada.
  • Auditoría desacoplada.

🧩 Componentes mínimos de una arquitectura híbrida (vista por capas)

Una arquitectura “bien pensada” se entiende en capas: interfaz, servicios, datos y seguridad. Así reduces el radio de daño si algo falla.

1) Capa de Aplicación

  • Web / Móvil / API Gateway
  • Validación de entrada y control de tasa (rate limiting)
  • Separación de endpoints públicos vs internos

2) Capa de Servicios

  • Servicio de negocio (módulos por dominio)
  • Servicio de auditoría (eventos + evidencias)
  • Cola de mensajes para tareas pesadas

3) Capa de Datos

  • SQL (privado): entidades, usuarios, transacciones internas
  • Objeto/Blob: adjuntos y evidencias (cifrado)
  • Blockchain: anclaje de pruebas (hashes), no datos sensibles

4) Capa de Seguridad

  • IAM (roles, permisos, MFA)
  • Gestión de secretos (KMS/Vault)
  • Monitoreo y alertas (SIEM / logs)
Regla práctica: si algo contiene PII, secretos o información operativa crítica, no va a blockchain pública. Se guarda en almacenamiento privado y se ancla su hash.
Privacidad primero Hash como prueba Minimización Trazabilidad

🔁 Flujo de datos recomendado (de “evento” a “prueba”)

Este flujo te ayuda a diseñar una solución que sea auditable y eficiente sin sacrificar privacidad.

1) La app genera un EVENTO (ej. “se aprobó solicitud #123”) 2) Se arma un paquete de evidencia: payload + metadata + timestamp + actor 3) Se calcula HASH = SHA-256(evidencia_normalizada) 4) Se guarda evidencia completa en almacenamiento privado (cifrada) + control de acceso 5) Se ancla el HASH en blockchain (prueba pública de integridad / fecha) 6) Se registra el vínculo: evento → hash → ubicación de evidencia → responsable → retesteo
Mini‑reto (15 min): elige un evento de tu sistema (registro, pago, aprobación, entrega) y define:
  • ¿Qué campos van en la evidencia? (mínimos)
  • ¿Qué se cifra y dónde se guarda?
  • ¿Qué se ancla en blockchain? (solo el hash)
  • ¿Quién puede verificar y con qué permiso?

🛡️ Controles de seguridad por diseño (checklist)

  • Gestión de claves: llaves fuera del código (KMS/Vault), rotación y privilegio mínimo.
  • Firmas y permisos: firma de transacciones con roles claros; evita “admin omnipotente”.
  • Observabilidad: logs estructurados + correlación (traceId) + alertas por anomalías.
  • Separación de ambientes: dev/test/prod con credenciales distintas y datos sintéticos.
  • Auditoría del smart contract: revisión de lógica, controles de acceso y eventos.
  • Plan de respuesta: qué hacer si se filtra un secreto o si hay un bug crítico (pausa/rotación/retest).

📦 Entregables que demuestran arquitectura (lo que evalúa un auditor)

Documentación

  • Diagrama de componentes + flujo de datos
  • Modelo de amenazas (qué puede fallar y cómo lo contienes)
  • Matriz de riesgos + controles

Evidencia técnica

  • Checklist de IAM (roles, MFA, permisos)
  • Plan de despliegue (testnet → auditoría → mainnet)
  • Registro de pruebas (hashes + ubicación cifrada)
