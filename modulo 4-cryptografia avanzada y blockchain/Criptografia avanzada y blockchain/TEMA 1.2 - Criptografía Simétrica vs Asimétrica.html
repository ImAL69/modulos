TEMA 1.2 - Criptografía Simétrica vs Asimétrica

Semana 1 • Comparativa técnica, uso híbrido y decisiones de diseño en sistemas reales

La criptografía simétrica y la asimétrica no compiten: se complementan. En arquitectura moderna se usan juntas para conseguir eficiencia (simétrica) y distribución segura de claves + identidad (asimétrica). Diseñar bien exige entender trade‑offs: rendimiento, latencia, tamaño de claves, superficie de ataque, y requisitos de cumplimiento.

En este tema revisaremos algoritmos comunes, casos de uso y decisiones de diseño típicas en sistemas reales (APIs, móviles, nube, IoT, banca, mensajería segura).

1) Criptografía simétrica: qué es y cuándo usarla
  • Idea central: una sola clave compartida cifra y descifra.
  • Fortaleza: es muy rápida (ideal para grandes volúmenes).
  • Reto: distribución de la clave (¿cómo la compartes sin que la roben?).
  • Modos recomendados: AEAD como AES‑GCM o ChaCha20‑Poly1305 (cifrado + autenticación).
  • Errores frecuentes: reutilizar nonce/IV en GCM/ChaCha20; usar ECB; usar CBC sin MAC.
2) Criptografía asimétrica: qué aporta realmente
  • Idea central: par de claves (pública/privada).
  • Usos clave: intercambio de claves (p.ej. ECDH), firmas digitales, certificados, identidad.
  • Coste: operaciones más lentas; no se usa para cifrar “todo” el tráfico.
  • Algoritmos: RSA (histórico y común), ECC (moderno y eficiente), Ed25519 (firmas rápidas).
  • Buenas prácticas: usar padding/formatos seguros (RSA‑PSS para firma, OAEP para cifrado de claves).

4) Cifrado híbrido: el patrón dominante

Patrón: usar asimétrica para acordar una clave de sesión y luego usar simétrica para cifrar el tráfico. Así se obtiene identidad, confidencialidad e integridad con rendimiento.

  • Ejemplo (TLS 1.3): ECDHE para secreto compartido + AEAD para datos + certificados para autenticar servidor.
  • Perfect Forward Secrecy: claves efímeras (ECDHE) reducen impacto si una clave privada se filtra luego.
  • Rotación de claves: sesiones cortas y re‑handshake controlado para minimizar exposición.
5) Casos reales y decisiones típicas
  • API/Backend: TLS obligatorio + tokens firmados; cifrado en reposo para datos sensibles + KMS.
  • Mensajería: doble ratchet (Signal) = asimétrica para establecer + simétrica con rotación constante.
  • IoT: ECC por eficiencia; claves por dispositivo; actualización segura (firmas) para firmware.
  • Blockchain: firmas (ECDSA/EdDSA) para autorizar; hashes para integridad; no es “cifrado de datos” por defecto.
  • Almacenamiento: simétrica (AES‑XTS para discos); asimétrica para envolver claves (key wrapping).

ParteMecanismoNota de implementación
EnvíoClave de sesión simétrica por archivo (AEAD)Genera nonce único; adjunta etiqueta de autenticación.
CompartirEnvolver clave con pública del destinatario (asimétrica)Usa ECIES/NaCl box o RSA‑OAEP (según stack).
IdentidadCertificados/firmasFirma metadata (hash del archivo + permisos) para no repudio.

Apéndice 1: Notas profesionales y ejemplos adicionales

Este apéndice amplía el tema con recomendaciones, contra‑ejemplos y prácticas de ingeniería para entornos reales (empresas, nube, DevSecOps y cumplimiento). Úsalo como material de consulta y para profundizar tu comprensión.

  • Recomendación práctica: documenta supuestos, amenaza principal y alcance (qué proteges y de quién).
  • Checklist de implementación: no inventes criptografía; usa librerías auditadas y modos autenticados (AEAD).
  • Gestión de claves: separa claves de datos, rota, revoca y registra accesos; usa KMS/HSM cuando aplique.
  • Validación: prueba con vectores oficiales (NIST/ISO) y añade pruebas negativas (inputs corruptos).
  • Telemetría segura: registra eventos de seguridad sin filtrar secretos; usa redacción (redaction) y control de acceso.
  • Riesgo humano: el error más frecuente no es el algoritmo, es el proceso (copiar/pegar claves, backups expuestos).
  • Mitigación: revisiones de código, análisis estático, y políticas de secretos (no en repositorios).
  • Cuando falle algo: ten plan de respuesta: rotación de claves, invalidación de tokens, comunicación y post‑mortem.

Mini‑caso: una API que cifra datos con AES‑CBC sin autenticación puede ser vulnerable a padding‑oracle. Solución: migrar a AES‑GCM o ChaCha20‑Poly1305 y aplicar versionado de esquema para transición segura.
