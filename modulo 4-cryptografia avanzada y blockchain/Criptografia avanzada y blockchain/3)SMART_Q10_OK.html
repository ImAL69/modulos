Tema 1.5: Ecosistema Blockchain y Smart Contracts

Tema 1.5 • Redes blockchain, smart contracts, plataformas, arquitectura y seguridad (Q10-safe)

En este tema profundizamos en el ecosistema blockchain y en el rol de los smart contracts como motores de automatización confiable. El objetivo es que puedas explicar y comparar tipos de redes (públicas, privadas, consorcio), entender cómo se despliegan contratos inteligentes, y reconocer riesgos de seguridad y decisiones de arquitectura que afectan costo, rendimiento, gobernanza y cumplimiento.

A diferencia de una base de datos tradicional, una blockchain combina: consenso (acuerdo sobre el estado), inmutabilidad práctica (historial verificable) y criptografía (hashes, firmas, pruebas). Los smart contracts agregan lógica programable que se ejecuta bajo reglas del protocolo.

1) Tipos de Blockchain y criterios de decisión

La categoría principal depende del control de acceso y del modelo de gobernanza. No existe “mejor” universal; depende del caso.

TipoAccesoVentajasLimitacionesEjemplos
PúblicaPermissionlessAlta transparencia, resistencia a censura, comunidad abiertaCosto variable, menor privacidad, throughput limitadoBitcoin, Ethereum, Solana
PrivadaPermissioned (1 org)Alto rendimiento, privacidad y controlCentralización, menor neutralidadHyperledger Fabric, Quorum
ConsorcioPermissioned (multi-org)Balance entre eficiencia y confianza compartidaGobernanza compleja, acuerdos B2B necesariosR3 Corda, redes sectoriales
  • Pregunta guía: ¿hay múltiples actores que no confían plenamente entre sí y necesitan un registro verificable común?
  • Si solo hay 1 actor: una BD + firma + auditoría suele ser más simple y barata.
  • Privacidad: evalúa qué datos van on-chain vs off-chain; en muchos casos se almacenan hashes y se custodian datos fuera.
2) Smart Contracts: definición, alcance y límites

Smart contract es un programa desplegado en una red blockchain que ejecuta reglas de negocio de forma determinística cuando se cumplen condiciones. Su fuerza está en la ejecución verificable y en la automatización.

  • No es “legal automático”: el contrato inteligente no reemplaza un contrato jurídico; lo complementa (prueba, ejecución, registro).
  • Determinismo: el mismo input debe producir el mismo output en todos los nodos.
  • Inmutabilidad: desplegado no se “edita” como un backend; se versiona o se usan patrones de upgrade controlados.
  • Costos: ejecutar lógica en cadena tiene costo (gas, fees). Optimizar storage y loops es esencial.

Modelo mental: máquina de estados + reglas + eventos para auditoría.

3) Plataformas y ejecución (EVM y alternativas)

Las plataformas difieren en su máquina virtual, lenguaje, costos y modelo de consenso. Esto impacta rendimiento, tooling y seguridad.

PlataformaLenguajes comunesPunto fuerteRiesgo/limitación típica
Ethereum (EVM)Solidity, VyperEcosistema y tooling masivo (DeFi/NFT)Costos altos en picos; complejidad de upgrades
SolanaRust/CAlto throughput y baja latenciaComplejidad de programación concurrente; historial de interrupciones
CardanoPlutus/HaskellEnfoque formal y académicoRitmo de adopción/desarrollo más lento
  • Tooling recomendado: frameworks (Hardhat/Foundry) + librerías auditadas (OpenZeppelin).
  • Seguridad: análisis estático (Slither) + fuzzing + auditorías independientes.
  • Observabilidad: indexación y alertas ante eventos críticos (pausas, upgrades, transfers anómalos).
4) Arquitectura de smart contracts (patrones)
  • Escrow: retiene fondos hasta condición (entrega/validación).
  • Tokenización: estándares ERC-20/721/1155; cuida permisos y supply.
  • Access Control: roles (admin, minter, pauser). Aplica mínimo privilegio.
  • Oráculos: puente con el mundo real (precio, eventos). Riesgo: manipulación y latencia.
  • Upgradability: proxy patterns. Riesgo: control centralizado si admin no está bien gobernado.
Checklist de diseño (antes de programar)
  • Define objetivos y amenazas: ¿quién puede atacar y qué gana?
  • Define invariantes (reglas que siempre deben cumplirse).
  • Define entradas externas: oráculos, firmas, timestamps. Minimiza dependencia.
  • Define operación: cómo se pausará/recuperará ante incidente (circuit breaker).
  • Define auditoría: eventos, monitoreo y runbook.
5) Seguridad en smart contracts (vulnerabilidades + mitigación)

En smart contracts, un bug puede significar pérdida irreversible. La seguridad debe planearse desde el diseño.

RiesgoQué pasaMitigación recomendada
ReentrancyLlamadas externas antes de actualizar estadoChecks-Effects-Interactions; reentrancy guard
Access Control débilFunciones críticas sin restriccionesRoles; revisiones; pruebas de permisos
Manipulación de oráculoDatos externos alteradosOráculos robustos; TWAP; validación cruzada
Overflow/UnderflowErrores aritméticosSolidity >=0.8; librerías seguras
Front-running/MEVOrden de tx explotadoCommit-reveal; límites; subastas; mitigación MEV
  • Testing: unit tests + fuzzing + escenarios adversariales.
  • Auditoría: revisión por pares + auditoría externa si mueve valor.
  • Operación: monitoreo de eventos (mints, upgrades, transfers) y alertas.
6) Casos de uso con criterios de valor
  • DeFi: préstamos, DEX, stablecoins. Valor: composabilidad. Riesgo: exploits/oráculos.
  • Supply Chain: trazabilidad. Valor: auditoría compartida. Datos sensibles off-chain.
  • Identidad: credenciales verificables. Valor: portabilidad, menor fricción.
  • NFT/propiedad: registro y royalties. Valor: autenticidad y transferibilidad.
  • Sector público: registros (con cuidado). Valor: verificabilidad; riesgo: privacidad.
Decisión: ¿Blockchain sí o no? (regla práctica)
  • Sí, si hay múltiples partes con baja confianza y se requiere verificabilidad compartida.
  • No, si una BD con auditoría/firma resuelve con menor costo y complejidad.
  • Evita on-chain datos personales; usa hashes y pruebas cuando aplique.
Apéndice profesional: roadmap de desarrollo seguro (paso a paso)
  1. Modelado: estados, actores, amenazas y flujos de valor.
  2. Diseño: invariantes, roles, eventos.
  3. Implementación: librerías auditadas; simplicidad.
  4. Pruebas: unit + fuzzing + ataques.
  5. Auditoría: peer review + externa.
  6. Despliegue: testnet → mainnet con monitoreo.
  7. Operación: alertas, respuesta a incidentes, rotación de llaves admin.
