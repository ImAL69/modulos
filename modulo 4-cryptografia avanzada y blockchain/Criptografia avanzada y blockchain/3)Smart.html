Tema 2.6: Introducción a Smart Contracts

Programación determinista e inmutable sobre blockchain.

INTRO

• ¿Qué es realmente un Smart Contract?

Un Smart Contract no es inteligente (no usa IA) ni es un contrato legal en sí mismo. Es un

programa informático

que se ejecuta de forma determinista en una blockchain.

Cada nodo de la red ejecuta exactamente el mismo código, con los mismos datos, y llega al mismo resultado. Esto elimina la necesidad de intermediarios.

DIFERENCIA

• Contrato legal vs Smart Contract

  • Contrato legal:

    lenguaje natural, interpretación humana, tribunales.

  • Smart Contract:

    código, reglas estrictas, ejecución automática.

En la práctica, los Smart Contracts

ejecutan acuerdos

, pero no reemplazan por completo al marco legal tradicional.

CÓDIGO BÁSICO

// Ejemplo básico en Solidity contract RegistroPropiedad { mapping(address => string) public propiedades; function registrar(string memory hashDoc) public { propiedades[msg.sender] = hashDoc; } }

Este contrato permite asociar un hash a una dirección blockchain. Una vez ejecutado, el registro queda almacenado de forma permanente.

ESTADO

🗄️ Estado del contrato

El estado es la información que queda guardada en la blockchain:

  • Mappings:

    relaciones clave-valor.

  • Structs:

    estructuras complejas.

  • Variables:

    contadores, flags, configuraciones.

Cada cambio de estado cuesta gas y queda registrado para siempre.

REGLA DE ORO

•

Regla de oro:

el código desplegado es

inmutable

. Si hay un error, el error se queda.

PATRÓN SEGURO

// Patrón recomendado (simplificado) require(permitido[msg.sender]); require(bytes(hashDoc).length == 64); propiedades[msg.sender] = hashDoc; emit Registrado(msg.sender, hashDoc, block.timestamp);

PENSAMIENTO

• Piensa siempre en: Estado + Reglas + Eventos

  • Estado:

    qué información se guarda.

  • Reglas:

    quién puede modificarla.

  • Eventos:

    evidencia auditable y trazable.

GAS

⛽ Gas y costos

Cada operación tiene un costo computacional (gas).

  • Loops grandes → alto consumo.

  • Escrituras en storage → costosas.

  • Mal diseño → contratos inviables.

Optimizar gas no es solo eficiencia: es seguridad económica.

VULNERABILIDADES

🐞 Vulnerabilidades típicas

  • Reentrancy:

    llamadas externas antes de actualizar estado.

  • Access control:

    funciones críticas públicas.

  • Integer overflow:

    errores aritméticos.

  • Dependencia de oráculos:

    datos externos manipulables.

Mitigaciones:

checks-effects-interactions, roles, pausas, auditorías.

CASOS USO

💡 Casos de uso adecuados

  • Escrow y pagos condicionados.

  • Registro público de hashes.

  • Votaciones simples y verificables.

  • Automatización de reglas claras.

CUANDO NO

⛔ Cuándo NO usar Smart Contracts

  • Cuando los datos son privados.

  • Cuando las reglas cambian frecuentemente.

  • Cuando el costo supera el beneficio.

  • Cuando una base de datos es suficiente.

Complemento: Smart Contracts con enfoque de seguridad

  • Regla de oro:

    el contrato desplegado es difícil de cambiar; diseña para minimizar errores.

  • Estado + reglas + eventos:

    el estado guarda datos; reglas restringen cambios; eventos dejan evidencia auditable.

  • Control de acceso:

    roles, modifiers, “least privilege”; evita funciones críticas públicas.

  • Riesgos típicos:

    reentrancy, integer issues, dependencia de oráculos, front‑running.

  • Mitigaciones:

    checks‑effects‑interactions, pausas, límites de gas, auditorías.

Checklist para entregar un contrato simple

  • Define qué se guarda (mínimo necesario).

  • Define quién escribe y con qué permisos.

  • Emite eventos con timestamp/tx.

  • Incluye validaciones de entrada (longitud/formato).

  • Prueba casos de error (entrada inválida, rol no permitido).

Optimizado para Q10: sin scripts, sin estilos externos y sin dependencias bloqueables.

Complemento profesional: Smart Contracts seguros

  • Estado + reglas + eventos:

    guarda lo mínimo, valida todo, emite eventos auditables.

  • Acceso:

    roles/modifiers y mínimo privilegio.

  • Riesgos:

    reentrancy, front‑running, oráculos, bugs inmutables.

  • Mitigaciones:
