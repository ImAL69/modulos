Reglas del Laboratorio

Ética y responsabilidad en el uso de criptografía y blockchain

Un entorno de aprendizaje seguro requiere responsabilidad. Estas reglas protegen tanto a los estudiantes como a los sistemas.

Importante: La criptografía y blockchain son herramientas poderosas que pueden usarse tanto para proteger como para atacar sistemas. Este laboratorio es un entorno educativo donde la responsabilidad y ética son fundamentales. El incumplimiento de estas reglas puede resultar en consecuencias académicas y legales.

1. Manejo Seguro de Claves Privadas ALTA GRAVEDAD

NUNCA compartas, subas a la plataforma ni almacenes en repositorios públicos claves privadas reales. En los laboratorios, usaremos exclusivamente claves de prueba generadas específicamente para cada ejercicio.

¿Por qué es tan importante?
  • Una clave privada comprometida significa pérdida total de control sobre los activos protegidos
  • En blockchain, perder una clave privada significa perder acceso permanente a los fondos
  • Las claves expuestas pueden ser robadas por bots que escanean repositorios públicos
  • Subir claves reales viola políticas de seguridad institucionales y puede tener consecuencias legales
Alternativas seguras:
  • Usar variables de entorno (no commitear archivos .env)
  • Implementar sistemas de gestión de secretos (Vault, AWS Secrets Manager)
  • Usar claves de prueba con prefijos claros como "TEST_" o "DEMO_"
  • Documentar en README que se usan claves de prueba, no reales

2. Prohibición Absoluta de Cifrado Malicioso ALTA GRAVEDAD

El uso de ransomware, cifrado malicioso o herramientas diseñadas para atacar sistemas ajenos está terminantemente prohibido. Las herramientas de cifrado deben usarse exclusivamente para proteger datos propios o de prueba en entornos controlados.

Prohibido específicamente:
  • Desarrollar, probar o implementar ransomware
  • Cifrar archivos de sistemas que no te pertenecen (laboratorios, servidores compartidos)
  • Usar herramientas de cifrado para extorsión o chantaje
  • Probar herramientas de ataque en sistemas de producción o de otros estudiantes
¿Qué está permitido?
  • Cifrar tus propios archivos de prueba en tu máquina local
  • Implementar sistemas de cifrado para proteger datos legítimos
  • Estudiar técnicas de ataque en entornos aislados (máquinas virtuales, contenedores)
  • Participar en CTFs (Capture The Flag) autorizados con fines educativos

3. Uso Exclusivo de Redes de Prueba (Testnet) MEDIA GRAVEDAD

Para todas las prácticas de Blockchain, utilizaremos exclusivamente redes de prueba (Testnets) o simuladores locales. No se permite realizar transacciones con dinero real en redes principales (Mainnet) durante las actividades del curso.

Redes de prueba recomendadas:
  • Bitcoin: Testnet3, Regtest, Signet
  • Ethereum: Goerli, Sepolia, Holesky, Ganache (local)
  • Hyperledger: Fabric test networks, Besu local
  • Solana: Devnet, Testnet, Local validator
Razones para usar testnets:
  • Evitar pérdidas financieras por errores en smart contracts
  • Permitir experimentación sin costo
  • Proteger la privacidad (transacciones en mainnet son públicas)
  • Cumplir con regulaciones de instituciones educativas

4. Protección de Datos Sensibles MEDIA GRAVEDAD

Evita completamente subir capturas, código o documentos que contengan datos sensibles reales. Esto incluye claves privadas, frases semilla (seed phrases), tokens de API, credenciales, correos personales o datos de identificación reales.

Técnicas de ofuscación permitidas:
  • Para capturas: Pixelar/blur áreas sensibles, usar editores para tapar información
  • Para código: Usar variables de entorno, archivos de configuración excluidos de git
  • Para datos de prueba: Usar datos ficticios (test@ejemplo.com, 12345678A)
  • Para tokens: Mostrar solo primeros y últimos caracteres (sk_live_***abc123)
Ejemplos de datos que NUNCA deben exponerse:
  • Claves privadas de wallets de criptomonedas
  • Frases de recuperación (12/24 palabras)
  • Tokens de API de servicios en la nube (AWS, Google, Azure)
  • Credenciales de bases de datos o servidores
  • Documentos de identidad personales (DNI, pasaporte)

5. No Inventes Criptografía ALTA GRAVEDAD

En prácticas y proyectos, usa exclusivamente librerías criptográficas estándar y probadas. Implementar algoritmos criptográficos "desde cero" o modificar algoritmos existentes suele introducir vulnerabilidades graves.

Librerías recomendadas:
  • Python: cryptography, PyNaCl (libsodium), hashlib (estándar)
  • JavaScript/Node.js: Web Crypto API, crypto (estándar), libsodium.js
  • Java: Bouncy Castle, Java Cryptography Architecture
  • C/C++: OpenSSL, libsodium, Crypto++
  • Go: crypto estándar, golang.org/x/crypto
Por qué NO implementar desde cero:
  • Los algoritmos requieren revisión por pares durante años
  • Implementaciones caseras tienen vulnerabilidades de tiempo (timing attacks)
  • Errores en generación de números aleatorios comprometen toda la seguridad
  • Las librerías estándar reciben auditorías continuas
¿Qué SÍ puedes implementar?
  • Integración de librerías criptográficas en tus aplicaciones
  • Protocolos que usan criptografía (ej: autenticación con JWT)
  • Sistemas de gestión de claves (usando librerías existentes)
  • Aplicaciones que consumen APIs criptográficas

Mejores Prácticas Recomendadas

Sigue estas prácticas para mantener un entorno de aprendizaje seguro y efectivo:

Ambientes Aislados

Usa máquinas virtuales, contenedores Docker o entornos cloud aislados para experimentación riesgosa.

Control de Versiones

Usa .gitignore para excluir archivos sensibles. Nunca commitees claves, tokens o configuraciones privadas.

Documentación Clara

Documenta claramente cuando uses datos de prueba o entornos simulados para evitar confusiones.

Colaboración Responsable

Comparte conocimientos, no vulnerabilidades. Reporta problemas de seguridad a instructores, no los explotes.

6. Prácticas Seguras con Blockchain MEDIA GRAVEDAD

Trabaja exclusivamente con redes de prueba, simuladores locales o sandboxes autorizados. No compartas wallets reales, no realices transacciones con dinero real durante el curso, y no participes en actividades que puedan afectar redes reales.

Prácticas prohibidas en el contexto educativo:
  • Conectar wallets de mainnet a aplicaciones de prueba
  • Realizar transacciones financieras reales como parte de ejercicios
  • Participar en DeFi (Finanzas Descentralizadas) con fondos reales
  • Interactuar con smart contracts en mainnet sin supervisión
  • Minar criptomonedas en hardware institucional
Entornos seguros para aprendizaje:
  • Ganache: Simulador local de Ethereum
  • Hardhat Network: Entorno de desarrollo Ethereum
  • Truffle Develop: Blockchain local para testing
  • Fabric Test Network: Para Hyperledger Fabric
  • Sandboxes de proveedores cloud: AWS Blockchain Templates, Azure Blockchain Workbench

Consideraciones Legales y Éticas

Marco legal aplicable: Las actividades en este laboratorio están sujetas a leyes de ciberseguridad, propiedad intelectual y protección de datos. En muchos países:

  • El acceso no autorizado a sistemas informáticos es un delito
  • El desarrollo o distribución de malware (incluyendo ransomware) es ilegal
  • La exposición negligente de datos personales puede violar regulaciones como GDPR
  • Las actividades con criptomonedas pueden estar reguladas por leyes financieras

Responsabilidad estudiantil: Como estudiante, eres responsable de:

  • Conocer y cumplir las políticas de uso aceptable de la institución
  • Reportar vulnerabilidades descubiertas a los instructores, no explotarlas
  • Proteger la confidencialidad de información institucional y de compañeros
  • Usar el conocimiento adquirido éticamente en tu futura carrera profesional

La educación en ciberseguridad viene con la responsabilidad de usar ese conocimiento para proteger, no para atacar.

Compromiso Ético y Responsable

Al participar en este módulo, reconozco que la criptografía y blockchain son herramientas poderosas que requieren responsabilidad. Me comprometo a:

  • Usar estas tecnologías exclusivamente para fines educativos y de protección
  • Respetar la privacidad y seguridad de sistemas ajenos
  • Seguir las mejores prácticas de seguridad en todas mis actividades
  • Reportar cualquier problema de seguridad descubierto a los instructores
  • Aplicar este conocimiento éticamente en mi desarrollo profesional

Este compromiso no requiere firma física, sino responsabilidad constante. La verdadera aceptación se demuestra mediante acciones responsables durante todo el curso.
