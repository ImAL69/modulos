Tema 2.6: Introducción a Smart Contracts

Lógica programable en la red.

Un Smart Contract no es "inteligente" (no usa IA) ni es un "contrato" legal per se. Es un script que se ejecuta en la blockchain.

// Ejemplo simplificado en Solidity
contract RegistroPropiedad {
  mapping(address => string) public propiedades;

  function registrar(string memory hashDoc) public {
    propiedades[msg.sender] = hashDoc;
  }
}

Este código permite que cualquier usuario (msg.sender) asocie un hash de documento a su dirección de forma permanente.

Riesgo de Seguridad: El código en blockchain es inmutable. Si subes un contrato con un error (bug), no puedes parchearlo fácilmente. El dinero o los datos atrapados en un contrato defectuoso pueden perderse para siempre.
Regla de oro: un Smart Contract es código inmutable. Si te equivocas, el error se queda.
// Patrón seguro (idea): validar + actualizar estado + emitir evento
require(permitido[msg.sender]);
require(bytes(hashDoc).length == 64);
propiedades[msg.sender] = hashDoc;
emit Registrado(msg.sender, hashDoc, block.timestamp);
Piensa en “Estado + Reglas + Eventos”
  • Estado: qué guarda el contrato (mappings, structs).
  • Reglas: quién puede hacer qué y cuándo.
  • Eventos: registro auditado (logs) para trazabilidad.
Vulnerabilidades típicas (para que no te tumben)
  • Reentrancy: llamada externa antes de actualizar estado.
  • Control de acceso pobre: funciones críticas públicas.
  • Oráculos manipulables: datos externos falsos (precio, identidad, eventos).
  • DoS por gas: loops grandes y operaciones costosas.

Mitigaciones: checks-effects-interactions, roles, pausas, límites, auditoría.

Casos de uso “buenos”
  • Escrow/depósito en garantía (liberar fondos con condición verificable).
  • Registro de certificados por hash (verificación pública).
  • Seguimiento de cadena de suministro con eventos por etapa.
