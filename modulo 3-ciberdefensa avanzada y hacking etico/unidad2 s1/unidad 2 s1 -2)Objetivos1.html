Tema 2.1: Metodologías de Pentesting

Estándares globales para una ejecución profesional y auditable.

Un "ataque" desordenado no sirve de nada. Para que un pentest tenga validez legal y técnica, debe seguir una metodología estandarizada. No se trata de "ver qué encuentro", sino de seguir un proceso científico y reproducible.

PTES (Penetration Testing Execution Standard)

Es la "biblia" de la ejecución. Cubre el ciclo completo: desde la negociación comercial (Pre-engagement) hasta la inteligencia, modelado de amenazas, análisis de vulnerabilidad, explotación, post-explotación y reporte.

Estándar de facto

OWASP (Open Web Application Security Project)

El estándar absoluto para seguridad web. Se enfoca en vulnerabilidades de aplicaciones (SQL Injection, XSS, Broken Auth). Si vas a auditar una web, DEBES usar la OWASP Testing Guide.

Web & Apps

OSSTMM (Open Source Security Testing Methodology Manual)

Se centra en métricas científicas. No dice "esto es seguro", sino que calcula un "Risk Assessment Value" (RAV) basado en hechos verificables (visibilidad, acceso, confianza).

Métricas & Ciencia

NIST SP 800-115

La guía del gobierno de EE.UU. Es el estándar requerido para auditorías en sector gobierno y financiero. Es menos técnica y más procedimental.

Gobierno & Cumplimiento

Cómo elegir una metodología (sin confundirte)

No hay “una mejor” para todo: depende del objetivo, el tipo de sistema y el entregable esperado.

Guía rápida de decisión

  • Si el foco es web/app → prioriza OWASP (Testing Guide + Top 10).
  • Si necesitas ciclo completo “end‑to‑end” → PTES.
  • Si te piden métricas y medición → OSSTMM.
  • Si hay cumplimiento/sector público → NIST SP 800‑115.

Entregables típicos (lo que se espera recibir)

  • Alcance + RoE firmado (o aprobado) antes de ejecutar.
  • Registro reproducible (herramienta + comando + resultado).
  • Reporte técnico con evidencia y recomendación.
  • Resumen ejecutivo (impacto y prioridad en lenguaje de negocio).
Tip profesional: Puedes combinar metodologías: usar PTES como “esqueleto” y OWASP como profundidad para pruebas web.

Comparativo práctico

Marco ¿Cuándo usarlo? Fortaleza Limitación común
PTES Pentest completo (infra/app) con reporte formal Ciclo end‑to‑end, muy aplicable No detalla tanto pruebas web como OWASP
OWASP Apps web/APIs, autenticación, sesiones, inyecciones Profundidad técnica en web No cubre por sí sola todo el ciclo del pentest
OSSTMM Cuando debes medir y justificar con métricas Enfoque científico verificable Menos “paso a paso” práctico para principiantes
NIST 800‑115 Gobierno, compliance, entornos regulados Procedimiento y gobernanza Puede sentirse menos técnico/operativo
