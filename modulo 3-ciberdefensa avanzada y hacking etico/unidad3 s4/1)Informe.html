Recurso: Estructura del Informe Técnico

Guía de contenido para el entregable "Informe Técnico Detallado".

Este documento es para el personal de TI (SysAdmins, Desarrolladores). Debe ser lo suficientemente detallado para que puedan reproducir el ataque y aplicar la solución sin preguntarte nada.

1. Información del Hallazgo
  • ID: Código único (ej. VULN-001).
  • Nombre: Descriptivo (ej. "Inyección SQL en Formulario de Login").
  • Activo Afectado: URL, IP o Servidor específico.
  • Severidad (CVSS): Score v3.1 (Base, Temporal y Ambiental si aplica).
2. Descripción y Prueba de Concepto (PoC)

El "paso a paso" para replicar el fallo:

  • Herramientas: ¿Qué software usaste? (Burp Suite, Nmap, script python).
  • Pasos de Reproducción: 1. Ir a la URL... 2. Ingresar payload... 3. Observar error.
  • Evidencia: Capturas de pantalla OBLIGATORIAS mostrando el resultado.
3. Recomendación Técnica

La solución concreta:

  • Solución Inmediata: ¿Qué hacer ya? (ej. Sanitizar inputs, actualizar librería).
  • Solución a Largo Plazo: (ej. Implementar WAF, capacitación de código seguro).
  • Referencias: Enlaces a documentación oficial o CVEs.
4. Análisis de Impacto (Lenguaje Técnico + Negocio)

Explica qué podría pasar si el hallazgo se explota, en dos niveles:

  • Impacto técnico: fuga de datos, ejecución remota, escalamiento de privilegios, denegación de servicio, etc.
  • Impacto de negocio: caída de ventas, sanciones por datos, interrupción operativa, reputación, costos de recuperación.
  • Condiciones de explotación: ¿requiere autenticación? ¿se puede hacer remoto? ¿hay mitigaciones existentes?
5. Priorización (Qué arreglar primero y por qué)

Recomendación práctica: prioriza con una mezcla de severidad + explotabilidad + criticidad del activo.

  • P1 (Inmediato): RCE, auth bypass, exposición de datos sensibles, credenciales por defecto.
  • P2 (Alta): XSS almacenado, configuraciones débiles, rutas administrativas expuestas.
  • P3 (Programable): versiones con riesgo bajo, hardening recomendado, mejoras de logging.

Tip: Si dos hallazgos tienen CVSS similar, arregla primero el que tenga mayor impacto real (datos críticos o sistemas core).

6. Retest y Criterio de Cierre
  • Qué se prueba: el mismo vector de ataque que funcionó, con evidencia antes/después.
  • Qué NO se asume: “ya actualizamos” no es evidencia; se confirma con retest.
  • Salida esperada: el exploit ya no funciona y la configuración/versión evidencia la corrección.
7. Anexo de Evidencias (Formato recomendado)

Ordena tus capturas y logs para que el equipo de TI pueda seguirlas sin preguntarte:

  • E-01: Recon/Enumeración (puertos, endpoints, versión).
  • E-02: PoC (payload/steps + resultado).
  • E-03: Impacto (prueba controlada: lectura de dato, acceso limitado, etc.).
  • E-04: Remediación (cambio aplicado o comandos sugeridos).
  • E-05: Retest (antes/después).
