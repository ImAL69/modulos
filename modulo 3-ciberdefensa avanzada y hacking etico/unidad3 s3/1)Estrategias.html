Tema 3.3: Estrategias de Mitigación

Soluciones técnicas para cerrar las brechas encontradas.

No basta con decir "arréglalo". Debes decir cómo. Existen cuatro estrategias principales para mitigar riesgos técnicos:

  • 1. Hardening (Endurecimiento)

    Configurar el sistema para ser seguro por defecto. Cerrar puertos innecesarios, deshabilitar servicios no usados, eliminar usuarios "guest" y aplicar políticas de contraseñas robustas.

  • 2. Gestión de Parches (Patch Management)

    La cura definitiva para vulnerabilidades de software (CVEs). Implica actualizar sistemas operativos, frameworks y librerías de terceros a sus versiones seguras más recientes.

  • 3. Segmentación de Red

    Evitar el movimiento lateral. Si un atacante compromete el servidor web en la DMZ, no debería poder saltar directamente a la base de datos interna. Usar VLANs y Firewalls internos.

  • 4. Controles Compensatorios

    Cuando no se puede parchear (ej. sistema legado crítico), se añade una capa extra de defensa, como un WAF (Web Application Firewall) o monitoreo intensivo.

Cómo elegir la estrategia correcta (guía rápida)

En un reporte profesional no basta con “recomendar”. Debes proponer una acción realista, alineada al riesgo y al contexto del activo (criticidad, ventana de mantenimiento, equipo responsable y restricciones).

  • Si el riesgo es crítico y explotable remotamente: prioriza parche + hardening (y un control compensatorio temporal).
  • Si el sistema es legado y no se puede actualizar: usa controles compensatorios (WAF/IPS, segmentación, allowlist, monitoreo).
  • Si el fallo es de configuración: aplica hardening (baseline CIS, cierre de puertos, TLS seguro, deshabilitar features).
  • Si el problema es “movimiento lateral”: segmentación, MFA, y separación de privilegios.

Ejemplos de mitigación (listo para copiar al informe)

Usa estas frases como base y ajusta con tu evidencia (servicio, versión, ruta, host y fecha):

Hardening

“Deshabilitar servicios no utilizados y cerrar puertos expuestos. Implementar política de contraseñas y eliminar cuentas por defecto. Aplicar baseline de configuración segura y verificar con revisión posterior.”

Patch Management

“Actualizar el componente afectado a una versión corregida según el aviso del proveedor. Validar dependencias y realizar pruebas en staging. Registrar cambio en control de versiones y programar retest.”

Segmentación

“Separar la capa web de la base de datos en redes distintas. Limitar el tráfico solo a puertos necesarios mediante reglas de firewall internas. Implementar allowlist y monitorear accesos.”

Controles compensatorios

“Implementar WAF/IPS con reglas para el patrón explotable. Aumentar logging y alertas, y restringir el acceso por IP/rol. Mantener como medida temporal mientras se planifica el reemplazo/actualización.”

Checklist de cierre (para tu evidencia)

  • La recomendación indica qué cambiar, dónde, quién lo ejecuta y cuándo.
  • Incluye si aplica: ventana de mantenimiento, rollback y pruebas post-cambio.
  • Define un criterio de “listo”: evidencia de configuración/parche + retest exitoso.
