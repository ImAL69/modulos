Tema 3.3: Estrategias de Mitigación

Soluciones técnicas para cerrar las brechas encontradas.

No basta con decir "arréglalo". Debes decir cómo. Existen cuatro estrategias principales para mitigar riesgos técnicos:

  • 1. Hardening (Endurecimiento)

    Configurar el sistema para ser seguro por defecto. Cerrar puertos innecesarios, deshabilitar servicios no usados, eliminar usuarios "guest" y aplicar políticas de contraseñas robustas.

  • 2. Gestión de Parches (Patch Management)

    La cura definitiva para vulnerabilidades de software (CVEs). Implica actualizar sistemas operativos, frameworks y librerías de terceros a sus versiones seguras más recientes.

  • 3. Segmentación de Red

    Evitar el movimiento lateral. Si un atacante compromete el servidor web en la DMZ, no debería poder saltar directamente a la base de datos interna. Usar VLANs y Firewalls internos.

  • 4. Controles Compensatorios

    Cuando no se puede parchear (ej. sistema legado crítico), se añade una capa extra de defensa, como un WAF (Web Application Firewall) o monitoreo intensivo.

Cómo elegir la estrategia correcta (guía rápida)

En un reporte profesional no basta con “recomendar”. Debes proponer una acción realista, alineada al riesgo y al contexto del activo (criticidad, ventana de mantenimiento, equipo responsable y restricciones).

  • Si el riesgo es crítico y explotable remotamente: prioriza parche + hardening (y un control compensatorio temporal).
  • Si el sistema es legado y no se puede actualizar: usa controles compensatorios (WAF/IPS, segmentación, allowlist, monitoreo).
  • Si el fallo es de configuración: aplica hardening (baseline CIS, cierre de puertos, TLS seguro, deshabilitar features).
  • Si el problema es “movimiento lateral”: segmentación, MFA, y separación de privilegios.

Ejemplos de mitigación (listo para copiar al informe)

Usa estas frases como base y ajusta con tu evidencia (servicio, versión, ruta, host y fecha):

Hardening

“Deshabilitar servicios no utilizados y cerrar puertos expuestos. Implementar política de contraseñas y eliminar cuentas por defecto. Aplicar baseline de configuración segura y verificar con revisión posterior.”

Patch Management

“Actualizar el componente afectado a una versión corregida según el aviso del proveedor. Validar dependencias y realizar pruebas en staging. Registrar cambio en control de versiones y programar retest.”

Segmentación

“Separar la capa web de la base de datos en redes distintas. Limitar el tráfico solo a puertos necesarios mediante reglas de firewall internas. Implementar allowlist y monitorear accesos.”

Controles compensatorios

“Implementar WAF/IPS con reglas para el patrón explotable. Aumentar logging y alertas, y restringir el acceso por IP/rol. Mantener como medida temporal mientras se planifica el reemplazo/actualización.”

Checklist de cierre (para tu evidencia)

  • La recomendación indica qué cambiar, dónde, quién lo ejecuta y cuándo.
  • Incluye si aplica: ventana de mantenimiento, rollback y pruebas post-cambio.
  • Define un criterio de “listo”: evidencia de configuración/parche + retest exitoso.

10. Kit de estrategias por categoría de riesgo (nivel PRO)

Este bloque cuatriplica valor: transforma conceptos en decisiones ejecutables por tipo de hallazgo (identidad, exposición, CVEs, configuración y procesos).

10.1 Identidad y cuentas (IAM + higiene)

Cuando el vector principal es identidad (password reuse, tokens filtrados, cuentas huérfanas), la mitigación más efectiva es MFA + mínimo privilegio + gestión de secretos, con auditoría y verificación.

MFA & Access Policies

Quick winExige MFA en privilegios, endurece recuperación de cuenta y aplica políticas por contexto (ubicación/dispositivo).

Re-test: - Login sin 2FA falla - Se registra challenge en logs - No hay bypass por endpoints alternos

Least Privilege

EstructuralRoles por función, separación de funciones, y revisión periódica de permisos para evitar “acceso por costumbre”.

Re-test: - Usuario estándar no ejecuta acciones admin - Se evidencia reducción de permisos en auditoría

Rotación y secrets

Quick winRota claves/tokens, elimina secretos en código y centraliza en un gestor de secretos.

Re-test: - Secret anterior invalida - Nuevo secret funciona - No hay secretos hardcodeados

10.2 Exposición y superficie de ataque (puertas abiertas)

Si el hallazgo depende de estar “público” (puertos, endpoints, paneles), la estrategia ganadora es cerrar/limitar + filtrar + monitorear para cortar el vector incluso antes del parche.

  1. Inventario: lista servicios expuestos y valida necesidad (principio de mínima exposición).
  2. Restricción: allowlist por IP/rol/segmento y ocultar rutas administrativas.
  3. Protección: reverse proxy/WAF, rate limit, bloqueo progresivo y headers seguros.
  4. Monitoreo: alertas por scanning, fuerza bruta, enumeración y anomalías.
Regla práctica: “No exponer” suele ser la mitigación más barata y con mayor impacto.

10.3 Vulnerabilidades de software (CVEs): flujo completo

Patch management profesional es proceso, no evento: triage → prueba → despliegue → retest. Esto evita “parches falsos” y reduce reincidencia.

Flujo recomendado: 1) Identificar versiones afectadas + alcance (activos críticos primero) 2) Evaluar explotabilidad (PoC público / explotación conocida / exposición) 3) Probar en staging + plan de rollback 4) Desplegar en ventana controlada 5) Retest + evidencia + cierre
EscenarioRiesgoMitigaciónEvidencia / Retest
CVE crítico en componente web expuestoRCE / toma de controlParche urgente + WAF temporalVersion update + bloqueo patrón
CVE medio en servicio internoEscalada con acceso previoParche en sprint + segmentaciónRe-scan interno + reglas
Dependencia vulnerable indirectaRiesgo “transitivo”Actualizar cadena + lockfilesSBOM/scan sin hallazgo

10.4 Configuración insegura (hardening con baseline)

La configuración insegura vuelve si no se estandariza. Convierte hardening en baseline, audítalo y automatiza donde puedas.

Checklist mínimo (servidores)
- Servicios mínimos y puertos cerrados - Logs centralizados + retención - Usuarios sin privilegios innecesarios - TLS moderno y cifrados fuertes - Backups verificados (restore test)
Checklist mínimo (apps web)
- Headers: HSTS, X-Content-Type-Options, CSP (según caso) - Rate limit en auth y endpoints sensibles - Mensajes uniformes (evitar enumeración) - Validación + encoding (inyecciones) - Sesión segura (cookies HttpOnly/SameSite)

10.5 Mini‑taller consultoría (45–60 min)

  • Selecciona 3 hallazgos (IAM, exposición, CVE).
  • Escribe recomendación PRO (qué/dónde/quién/plazo) para cada uno.
  • Define control compensatorio si el parche tarda.
  • Describe retest y evidencia esperada.
  • Ordénalos en matriz Impacto/Esfuerzo y arma un plan 30/60/90.

12. Estrategias por fase del ataque (Kill Chain / MITRE como guía práctica)

Una forma “consultoría” de escribir mitigación es mapearla contra fases: si cortas una fase temprana, evitas el incidente completo.

12.1 Reconocimiento & Enumeración

El atacante busca información para perfilar usuarios, endpoints y rutas. Tu objetivo es reducir señal y castigar abuso.

Mensajes distintos en loginEnumeración de usuariosMensajes uniformes + rate limitRespuestas idénticas + límites

12.2 Acceso inicial

Es donde “ganas” más barato: MFA, surface reduction, phishing-resistance, políticas de sesión y cierre de servicios expuestos.

Checklist acceso inicial: - MFA (preferible resistente a phishing) - Bloqueo progresivo / CAPTCHA adaptativo - Restricción de paneles y APIs sensibles - Validación de sesiones: expiración, rotación, revocación

12.3 Persistencia y escalada

Si el atacante entra, quiere quedarse y subir privilegios. Mitiga con segmentación, control de privilegios y detección (EDR/SIEM).

TécnicaIndicadorControl preventivoControl de detección
Credenciales localesAccesos rarosLeast privilegeAlertas por anomalías
Servicios nuevosPuertos inesperadosHardening baselineEDR + inventario

12.4 Exfiltración e impacto

Reduce impacto con controles de datos y resiliencia: clasificación, DLP (si aplica), backups probados, y planes de respuesta.

Checklist impacto: - Backups 3-2-1 + restore test - Retención de logs para investigación - Plan IR: roles, comunicación, contención - Controles de acceso a datos sensibles
