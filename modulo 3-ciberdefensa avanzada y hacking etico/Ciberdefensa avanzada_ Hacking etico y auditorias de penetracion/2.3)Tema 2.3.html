Tema 2.3: Validación de Vulnerabilidades

Confirmar el fallo con evidencia, sin destruir el sistema (y sin ruido).

Un escáner puede marcar “posible SQLi” o “XSS probable”. Un pentester profesional convierte esa señal en una conclusión: real o falso positivo, y lo demuestra con una PoC inocua + evidencia replicable. En este tema aprenderás un flujo de validación que prioriza seguridad, precisión y trazabilidad.

🧠 Mentalidad: validar ≠ explotar al máximo
  • Validar es confirmar una condición vulnerable con el mínimo impacto.
  • Explotar (a fondo) solo ocurre si está permitido, es necesario para demostrar impacto y sigue siendo controlado.
  • El objetivo profesional es reducir incertidumbre: menos ruido, más certeza.
✅ Regla práctica: “demuestra lo suficiente para justificar riesgo y remediación”.

Flujo profesional de validación (paso a paso)

Este flujo sirve para web, API, servicios de red y misconfigurations.

Fase Qué haces Qué buscas Evidencia mínima
1) Baseline Reproduces la petición/acción normal sin “payload”. Respuesta esperada (control). Captura/log con URL, parámetros y respuesta.
2) Hipótesis Defines qué variable debe cambiar si hay vulnerabilidad. Un indicador (tiempo, error, contenido, status). Nota breve: “si X cambia, entonces Y”.
3) PoC inocua Introduces un payload seguro, con mínimo impacto. Cambio consistente vs baseline. Request + response + timestamp.
4) Confirmación Repites variando un parámetro a la vez. Evitar coincidencias / flukes. 2–3 repeticiones registradas.
5) Impacto Demuestras impacto mínimo viable (sin exfiltrar masivo). Qué permite hacer un atacante. Captura con evidencia de alcance logrado.
6) Recomendación Defines corrección + re-test. Acción concreta y verificable. Texto: “cambiar X por Y” + cómo verificar.
🧪 PoC inocua: ejemplos seguros (solo laboratorio)

Una PoC inocua demuestra control sin causar daño. No necesitas “borrar”, “tumbar” o “extraer masivo”. Usa indicadores de confirmación (contenido, estado, tiempo) y mantén el impacto mínimo.

📌 Ejemplo conceptual: XSS (prueba inocua)

La señal de confirmación es que el navegador ejecuta código. Para laboratorio, el clásico indicador inocuo es un alert:

<script>alert('PoC: XSS validado en laboratorio')</script>

Qué documentar: el campo/parámetro vulnerable, el payload, y la captura donde se vea el alert (con URL visible).

✅ Importante: no usar scripts para robar cookies ni acciones dañinas. La PoC es demostración, no ataque real.

⏱️ Ejemplo conceptual: SQLi ciega (indicador por tiempo)

Cuando no puedes ver resultados, usas “timing” como indicador. Lo profesional es repetir y registrar tiempos:

Baseline: respuesta ~120ms
PoC: respuesta ~5120ms (repetido 3 veces)

Qué documentar: baseline, payload inocuo de tiempo, mediciones (3 repeticiones), y conclusión.

⚠️ Mantén límites de tasa y stop condition. Si hay degradación, paras y reportas.

Cómo reducir falsos positivos (lo que diferencia a un pro)

El 80% de la calidad viene de validar con criterio.

🔍 Señales típicas de falso positivo

  • El banner dice una versión, pero el servicio real está “detrás” (proxy/WAF).
  • La ruta existe, pero está protegida (401/403) y no es explotable.
  • El CVE aplica solo a una configuración específica que aquí no está.
  • El “error” no cambia con el payload (es un error genérico).
  • El hallazgo desaparece al repetir o depende de caché.

✅ Técnicas de confirmación

  • Comparar baseline vs PoC (antes/después) con un solo cambio.
  • Repetir 2–3 veces y registrar tiempos/estados.
  • Usar “controles negativos”: payload “seguro” que no debería activar nada.
  • Correlacionar con logs (si aplica) o con comportamiento consistente.
  • Explicar condiciones: “solo ocurre cuando…”.
⚖️ CVSS vs riesgo real (rápido y útil)

CVSS ayuda a estandarizar severidad técnica, pero el riesgo real depende de contexto: exposición, privilegios, datos afectados, controles compensatorios y valor del activo.

🧭 Criterios de detención (Stop conditions)
  • Caída del servicio / degradación visible.
  • Bloqueo por WAF/IDS o saturación (muchos 429/5xx).
  • Exposición accidental de datos sensibles (parar, avisar, no copiar masivo).
  • Cambio no previsto en configuración/sistema.
⚠️ En entornos reales, la RoE manda. En laboratorio, sigue la norma de “mínimo impacto”.

Validación de vulnerabilidades — versión “de campo” (nivel profesional)

Más que “probar payloads”: construir una conclusión defendible, reproducible y segura.

🎯 Resultado esperado (lo que un evaluador quiere ver)
  • Reproducción determinística: pasos claros para repetir el hallazgo sin “magia”.
  • Cadena de evidencia: request/response, tiempos, logs y contexto de autenticación.
  • PoC inocua: demuestra el fallo con el mínimo impacto (sin daño, sin exfiltrar datos sensibles).
  • Impacto razonado: qué podría pasar si un atacante lo explota (con supuestos explícitos).
  • Recomendación accionable: corrección + verificación posterior (“cómo sé que quedó bien”).
✅ Evidencia replicable 🧪 PoC mínima 🧾 Trazabilidad 🛡️ Seguridad operacional

Checklist pre-validación (5 minutos)

Antes de tocar el objetivo, define límites y reduce el riesgo de “romper” el entorno.

Alcance & autorización
  • ¿Estoy validando en ambiente permitido (lab / staging / objetivo autorizado)?
  • ¿Qué acciones están prohibidas (DoS, exfil, modificación de datos, brute force)?
  • ¿Tengo ventana de pruebas y contacto técnico si algo falla?
Controles de seguridad
  • Trabajo con cuentas de prueba y datos ficticios.
  • Registro todo: hora, IP, usuario, herramienta, payload mínimo.
  • Si hay riesgo, preparo plan de reversión (snapshot / rollback / backup).

Marco de validación (4 pasos que siempre aplican)

Paso Acción Pregunta clave Evidencia mínima
1) Control Ejecuta el flujo normal sin “payload”. ¿Cómo se ve la respuesta “sana”? Captura request/response + estado + tamaño de respuesta.
2) Perturbación Cambia una variable a la vez (parámetro, header, body). ¿Qué cambia cuando altero este input? Diferencias medibles: código, cuerpo, tiempo, headers.
3) Confirmación Usa una PoC inocua para demostrar condición vulnerable. ¿Puedo demostrarlo sin causar impacto? Reproducción 2/2 o 3/3 + evidencia consistente.
4) Descarte Prueba hipótesis alternativas (falsos positivos). ¿Podría ser caching, WAF, error genérico? Prueba A/B, variaciones, control negativo.
⚠️ Regla de oro: si no puedes reproducir de forma consistente y explicar por qué ocurre, no está validado.

Guía por familia de vulnerabilidad (PoC mínima + evidencia)

Abre cada sección y quédate con el patrón: control → perturbación → confirmación → descarte.

🧩 SQL Injection (SQLi) — confirmar sin “dump”

Qué valida: el input llega a una consulta SQL sin sanitización adecuada o sin parametrización.

PoC inocua (ideas)
  • Boolean-based: comparar respuesta de 1=1 vs 1=2.
  • Error-based (si aplica): provocar error controlado que revele sintaxis (sin datos).
  • Time-based: micro-retardo medible (con límites y pocas repeticiones).
Falsos positivos comunes
  • Errores genéricos del backend para cualquier input raro.
  • WAF que responde distinto por patrón (sin que llegue a DB).
  • Cache/CDN que altera respuestas.
✅ Evidencia: dos requests casi iguales que producen diferencia consistente (status/body/time) + explicación de por qué.
🪞 XSS (Reflected/Stored) — demostrar ejecución segura

Qué valida: el input se refleja o persiste y se interpreta como código en el navegador.

PoC inocua
  • Usa un payload que solo genere un marcador visual (sin robar cookies).
  • Preferible: prueba con contexto (HTML/atributo/JS) para evitar “falso ok”.
  • Captura: DOM/console + dónde quedó el input (fuente exacta).
Descarte
  • ¿Se codifica/escapa correctamente? (HTML entities)
  • ¿CSP bloquea la ejecución? (sigue siendo hallazgo, cambia impacto).
  • ¿Solo se ve texto? entonces es reflejo seguro, no XSS.
✅ Evidencia: input → punto de inyección → ejecución verificada + captura + paso a paso.
🧭 IDOR / Broken Access Control — “tengo acceso a lo que no debo”

Qué valida: modificando un identificador (ID, UUID, ruta) accedes a recursos de otro usuario/rol.

PoC inocua
  • Crear dos usuarios de prueba (A y B).
  • Capturar un recurso del usuario A (por ejemplo: /orders/123).
  • Intentar acceder con usuario B cambiando el ID.
Evidencia mínima
  • Request con sesión/Token de B.
  • Respuesta que entrega datos de A (o permite acción sobre A).
  • Comparación con control (B accediendo a su propio recurso).
⚠️ Evita exponer datos reales: usa datos ficticios o redacta capturas (masking).
🌐 SSRF — validar sin “pegarle” a Internet

Qué valida: el servidor hace peticiones a URLs controladas por el atacante (o internas).

PoC inocua
  • Usar un endpoint controlado en laboratorio (no un tercero).
  • Validar por logs del servidor destino (hit) o por respuesta.
  • Si aplica: probar acceso a metadata interna solo en entorno controlado.
Falsos positivos
  • El servidor valida URL pero no la solicita.
  • Solo hace fetch de imágenes con proxy seguro.
  • Bloqueos por DNS o egress filtering.
✅ Evidencia: correlación entre tu request y un request saliente del servidor (timestamp + headers).

Matriz de evidencia (qué guardar y por qué)

Un reporte sólido se “defiende” con artefactos: no con opiniones.

Artefacto Qué contiene Para qué sirve Buenas prácticas
Request/Response URL, método, headers, body, status, tamaño Reproducción + comparación control vs PoC Masking de tokens y datos sensibles
Captura/Video Paso a paso visual (navegador/terminal) Demostrar flujo completo Mostrar reloj/tiempo si se usa time-based
Logs Eventos del servidor, auth, errores Confirmar causa raíz Conservar correlación (request id / trace id)
Entorno Versión, config relevante, rol/usuario Contexto para corregir y re-test Indicar supuestos (WAF, CDN, proxies)

Plantilla de reporte (lo mínimo que debe tener)

🧾 Estructura recomendada para el hallazgo (copiar/pegar)
Título: [Tipo] — [Recurso] — [Condición] Severidad: [Crítica/Alta/Media/Baja] | Confianza: [Alta/Media/Baja] Activo afectado: [URL/IP/Servicio] Descripción breve: (1–2 párrafos) Impacto: (qué permite y bajo qué supuestos) Pasos para reproducir: 1) Control: ... 2) PoC inocua: ... 3) Evidencia: ... Evidencias adjuntas: [capturas / requests / logs] Recomendación: - Fix inmediato: ... - Fix estructural: ... Validación de corrección (re-test): ...
✅ “Confianza” es clave: si no estás 100% seguro, dilo y explica qué falta para confirmarlo.

Mini‑reto profesional (para practicar)

🏁 Reto: convertir un hallazgo “ruidoso” en evidencia sólida
  1. El escáner reporta “Posible XSS” en un parámetro q.
  2. Define tu control (request normal) y documenta respuesta base.
  3. Haz 2 perturbaciones (cambia solo un elemento a la vez).
  4. Si confirmas, redacta el hallazgo con la plantilla y define el re-test.
  5. Si NO confirmas, escribe por qué: ¿código seguro? ¿WAF? ¿escape? ¿CSP?
