📘 Guía de Lectura: CVSS v3.1

Common Vulnerability Scoring System - Especificación

¡Bienvenido/a al mundo de la evaluación de vulnerabilidades! El CVSS (Common Vulnerability Scoring System) es un estándar abierto para asignar puntuaciones a las vulnerabilidades de seguridad, permitiendo a las organizaciones priorizar sus esfuerzos de remediación de manera objetiva. La versión 3.1 refina el modelo para capturar con mayor precisión las características intrínsecas de una vulnerabilidad.

Esta guía de lectura te ayudará a navegar por la estructura del estándar, entender sus tres grupos de métricas (Base, Temporal y Ambiental) y cómo se combinan para generar una puntuación numérica y una representación textual (vector).

⚙️ Parte 2: Métricas Base (Sección 2)

Estas métricas representan las características intrínsecas de una vulnerabilidad, constantes en el tiempo y en todos los entornos. Se dividen en dos subgrupos: Explotabilidad e Impacto.

🎯 Explotabilidad (2.1)
  • 2.1.1 Vector de Ataque (AV) - ¿Remoto, adyacente, local o físico?
  • 2.1.2 Complejidad de Ataque (AC) - ¿Condiciones especiales? (Baja/Alta)
  • 2.1.3 Privilegios Requeridos (PR) - ¿Se necesitan privilegios? (Ninguno/Bajo/Alto)
  • 2.1.4 Interacción de Usuario (UI) - ¿Requiere acción de un usuario? (Ninguna/Requerida)
💡 Ejemplo: Una vulnerabilidad que se explota remotamente sin credenciales (AV:N/PR:N) tendrá mayor puntuación.
🔄 Alcance (2.2)
  • 2.2 Scope (S) - ¿El impacto afecta a recursos más allá de la autoridad de seguridad del componente vulnerable? (Sin Cambio/Cambiado)
⚠️ Crítico: Un cambio de alcance (S:C) incrementa significativamente la puntuación.
💥 Impacto (2.3)
  • 2.3.1 Confidencialidad (C) - Alto/Bajo/Ninguno
  • 2.3.2 Integridad (I) - Alto/Bajo/Ninguno
  • 2.3.3 Disponibilidad (A) - Alto/Bajo/Ninguno
💡 Ejemplo: Pérdida total de confidencialidad (C:H) e integridad (I:H) será crítica.

⏳ Parte 3: Métricas Temporales (Sección 3)

Ajustan la puntuación Base según factores que cambian con el tiempo. Si no se especifican, se considera "No Definido" (X) y no afectan la puntuación.

💻 Madurez del Código Exploit (3.1)
  • Exploit Code Maturity (E) - ¿Existe código de exploit? (Alto/Funcional/Prueba de concepto/No probado/No definido)
🛡️ Nivel de Remedición (3.2)
  • Remediation Level (RL) - ¿Hay parche, workaround o nada? (No disponible/Solución temporal/Parche oficial/No definido)
✅ Confianza del Reporte (3.3)
  • Report Confidence (RC) - ¿Qué certeza hay de que la vulnerabilidad es real? (Confirmado/Razonable/Desconocido/No definido)
📌 Nota: La puntuación Temporal se calcula multiplicando la Base por los factores de estas métricas. Un exploit funcional (E:F) aumenta la puntuación; un parche oficial (RL:O) la disminuye.

🌍 Parte 4: Métricas Ambientales (Sección 4)

Permiten personalizar la puntuación al entorno específico de una organización, considerando la importancia de los activos y la existencia de controles mitigantes.

⚖️ Requisitos de Seguridad (4.1)
  • Confidentiality Requirement (CR) - Importancia de la confidencialidad (Alto/Medio/Bajo)
  • Integrity Requirement (IR) - Importancia de la integridad
  • Availability Requirement (AR) - Importancia de la disponibilidad
Re-ponderan las métricas de impacto según la criticidad del activo.
🔄 Métricas Base Modificadas (4.2)
  • MAV, MAC, MPR, MUI, MS, MC, MI, MA
Permiten sobrescribir las métricas Base si el entorno tiene características diferentes (ej. un servicio corre con menos privilegios).

🔤 Parte 6: Vector String (Sección 6)

Es la representación textual compacta de todas las métricas asignadas a una vulnerabilidad. Es la forma estándar de intercambiar información de CVSS.

Ejemplo: CVSS:3.1/AV:N/AC:L/PR:H/UI:N/S:U/C:L/I:L/A:N

Este vector indica: Red, Baja complejidad, Privilegios Altos, Sin interacción de usuario, Alcance sin cambio, Impacto Bajo en Confidencialidad e Integridad, y Ninguno en Disponibilidad.

📌 Orden: Aunque el orden preferido es Base → Temporal → Ambiental, los parsers deben aceptar cualquier orden.
